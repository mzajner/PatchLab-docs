## Documentation checks

- Frontmatter, route uniqueness, terminology rules, generated-file markers, registry count, and internal Markdown links are checked by `npm run check:content`.
- Astro type checks and a production static build are part of `npm run check`.
- The complete block reference is regenerated by `npm run sync:registry`; hand edits to generated pages are intentionally replaced.
- The desktop home page, introductory path, block reference, navigation density, and a compact viewport were visually reviewed from the production design.
- The current application was launched through its sanctioned review driver and compared with the interface terminology used in the docs.

## Intentionally incomplete areas

The initial public set does not invent instructions where release behaviour is still unsettled. Installation and distribution, complete Canvas authoring, Agent provider setup, export signing and notarization, DAW-specific hosting, and host-specific troubleshooting remain gated on stable product decisions and reproducible evidence. Their status is visible on the public documentation-status page and in the roadmap.

## Review rule for future changes

A fact is ready to publish only when it can be traced to at least one stable source: executable behaviour, a machine-readable product interface, a focused source inspection, or an approved release decision. DSP explanations additionally require dimensional checks, stated conventions, and a primary technical reference where a standard or equation is involved.
